### The Healthy Regime Is Clear

The strategy looks materially healthier in markets that remain two-sided longer.

T-120 strategy conditioning:
- **Still two-sided through T-60**: `692W / 1230L (36.0%)` | **+$21.63/trade** | **43.3% ROI** | avg ask `28.7¢`
- **Already one-sided by T-120**: `15W / 537L (2.7%)` | **+$5.00/trade** | **10.0% ROI** | avg ask `1.8¢`

Interpretation:
- One-sided-late markets are not the same regime as healthy two-sided markets
- They may still show nominal positive ROI in the simplified sim because prices are tiny, but they are clearly a much weaker and more fragile regime
- The best default filter direction is likely **favor markets that remain two-sided through at least T-60**

### Collector Enriched With Seven New Data Categories

Expanded the live collector to persist the additional market-structure data requested.

`src/scripts/pricing-collector.ts` now records:
1. **Book shape**
   - top 3 bid/ask levels per side
   - cumulative depth within `1¢`, `2¢`, and `5¢`
   - best-depth at underdog ask and favorite bid
2. **Slippage at size**
   - execution estimates for `$10`, `$25`, `$50`, `$100`
   - buy/sell average price, fill ratio, worst price, effective spread
3. **One-sided lifecycle**
   - two-sided / one-sided / untradable ratios
   - first and last times for each state
   - transition counts and one-sided reopen count
4. **Quote stability / churn**
   - quote changes
   - spread changes
   - underdog-side flips
   - largest underdog ask / favorite bid / favorite implied jumps
5. **Underlying price path**
   - running high/low
   - short-horizon moves (`30s`, `60s`, `120s`)
   - realized volatility
   - reversal-after-T120 flag
6. **Snapshot freshness / collection quality**
   - fetch latency
   - quote age from CLOB `timestamp`
   - capture delay vs target snapshot time
   - missed snapshot reasons
7. **Explicit regime labels**
   - `T-120` / `T-90` / `T-60` state
   - first tradable bucket
   - first one-sided bucket
   - collapse-before-T120 / collapse-before-T60 flags

Important limitation:
- The new book-shape, slippage, freshness, and missed-snapshot fields are only available going forward from the collector restart
- Older rows do not contain full ladders or quote timestamps, so those parts cannot be backfilled

## Session 4: Wallet Reverse-Engineering Pipeline

### Public Wallet Trade Source Was Identified

Tested several Polymarket trade-access paths before wiring any wallet research assumptions into the code.

Key validation result:
- authenticated CLOB `getTrades()` is **not** the right public market-tape source for this project
- public `getMarketTradesEvents()` did not return usable crypto trade data in live probing
- the usable source for crypto wallet research is the public Data API:
  - `https://data-api.polymarket.com/trades?market=<conditionId>`

New schema probe:
- `src/scripts/wallet-trade-schema-probe.ts`

Research spec:
- `docs/wallet-research-spec.md`

The probe confirmed public trade rows include:
- `proxyWallet`
- `name`
- `pseudonym`
- `transactionHash`
- `conditionId`
- `asset`
- `outcome`
- `side`
- `timestamp`

This was the critical prerequisite for pursuing reverse engineering instead of copy trading.

### Raw Wallet Trade Collection Was Added

Built a restart-safe raw collector around the public Data API:
- `src/scripts/wallet-trade-collector.ts`

Outputs:
- `wallet-trades.raw.jsonl`
- `state/wallet-trade-collector-state.json`

Collector behavior:
- tracks current/recent `btc/eth/sol/xrp` `5m/15m` markets from Gamma
- fetches public trade prints market-by-market
- appends raw-first records with wallet/profile metadata preserved
- dedupes by local collector state so reruns do not replay the same rows

Verification:
- first one-shot pass collected `18,223` raw wallet trades
- immediate rerun appended `0` rows for the same tracked windows
- later pass picked up the next set of windows correctly and expanded the raw file to `35,491` rows

### Wallet Trades Were Enriched Against Collector Regimes

Built a full rebuild enrichment layer:
- `src/scripts/enrich-wallet-trades.ts`

Output:
- `wallet-trades.enriched.jsonl`

Join logic:
- matches wallet trades to pricing collector records by market slug
- attaches liquidity profile, regime labels, quote/churn context, and nearest late-window snapshot
- only treats a snapshot as matched when the trade falls inside the observed collector window and within a `20s` timestamp tolerance
- also computes resolution-aware BUY-side proxy fields:
  - `resolvedOutcome`
  - `buyResolvedWin`
  - `buyHoldToResolutionPnl`

Current enriched dataset after rebuild:
- `35,491` enriched wallet trades
- `23,276` late-window matched trades
