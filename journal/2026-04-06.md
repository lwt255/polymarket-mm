### Critical Bug: Zombie Process Leak
The bot spawned hundreds of zombie `bullpen tracker trades` processes on VPS because:
1. Bullpen CLI calls hung when JWT expired (19:23 UTC)
2. `execSync` timeout doesn't kill child processes on Linux
3. VPS hit 1,101 tasks (max 1,110) and ran out of resources

**Fix**: Wrapped all Bullpen CLI calls with Linux `timeout` command + `killSignal: 'SIGKILL'`. Also had to re-login to get fresh JWT.

## Session 4: Lost Trade Data, JWT Hell, and the Pivot to Direct Polymarket API

### Lost 69 Trades — Stupid Mistake
Captured 69 dry-run trades, then **wiped them on every restart** by including `rm copy-trades.jsonl` in the restart command. User rightfully pissed. Saved a `feedback_never_delete_trade_data.md` memory so this never happens again. Bot now archives/preserves the ledger across restarts and deploys.

### JWT Refresh Rabbit Hole (Failed)
Spent hours trying to make Bullpen JWT auto-refresh work:
1. **JWT lifetime**: 15 minutes (not 3 hours like I assumed). Was confusing JWT exp with session/refresh token exp.
2. **Refresh endpoint exists**: `https://app.bullpen.fi/api/auth/refresh` accepts `{refresh_token}` POST
3. **Built auto-refresh** into bot with 5-min cooldown to avoid spamming
4. **All attempts returned HTTP 429** — refresh endpoint rate-limited persistently across logins, IPs, and accounts
5. **Tested CLI auto-refresh**: Confirmed `bullpen` CLI does NOT auto-refresh JWT on its own. It just hangs when JWT expires. This is a Bullpen CLI bug.
6. Required manual `bullpen login` (browser device flow) every ~15 minutes — completely unviable

### The Pivot — Direct Polymarket API (Massive Win)
User asked: "Do we actually need Bullpen for all this?"

**Answer: No.** Discovered Polymarket's data API is fully public with everything we need:

```bash
curl https://data-api.polymarket.com/trades?user=<addr>&limit=20
```

Returns: `proxyWallet`, `side`, `asset` (token ID), `conditionId`, `size`, `price`, `timestamp`, `title`, `slug`, `outcome`, `outcomeIndex`, `name`, **`transactionHash`**

This is **strictly better** than Bullpen's tracker:
- No auth required at all
- Includes data Bullpen doesn't expose (txhash, conditionId, asset)
- We can dedupe by tx hash (cleanest possible identifier)
- It's the actual source Bullpen wraps — cutting out the middleman

For execution, we already have a working CLOB client in this repo (`src/core/clob-client.ts` + `src/core/execution/order-executor.ts`) that uses our actual Polymarket EOA wallet — no separate "Bullpen wallet" to fund. The trades API gives us `asset` which is exactly the token ID `executeAndConfirm()` needs.

### v2 Bot Architecture
Rewrote `copy-trade-bot.ts` end-to-end:
- **Polling**: Direct calls to `data-api.polymarket.com/trades?user=<addr>` for each followed trader in parallel
- **Dedup**: By transaction hash (the cleanest unique key)
- **Execution**: Lazy-loaded CLOB client (only initialized in `--live` mode)
- **Discovery**: Periodic (6h) call to Bullpen leaderboard — only piece that still uses Bullpen, and it's unauthenticated so no JWT issues
- **Followed list**: Persisted to `state/copy-trade-followed.json`, can be edited dynamically
- **Ledger preservation**: Never deletes existing trade data; archives if needed

### Key Lessons
1. **Don't assume third-party wrappers are necessary** — always check if the underlying API is public
2. **Polymarket's data API is excellent** — fully public, comprehensive, no auth, no rate limit drama
3. **Build on infrastructure you control** — we already had CLOB execution working; piping it through Bullpen was a mistake
4. **Never delete trade data on restart** — even when "starting fresh", append-only with archive is the rule
