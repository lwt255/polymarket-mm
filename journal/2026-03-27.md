## Bot Development

### Built: Underdog Snipe Bot (`src/scripts/crypto-5min/underdog-snipe-bot.ts`)

**Architecture — safety-first design fixing the missed-trade bug:**

1. **Position Verifier** (`src/core/execution/position-verifier.ts`)
   - Reads on-chain USDC balance directly from Polygon
   - Hard max loss based on verified balance, not tracked P&L
   - Refuses to trade if balance read fails (no trading blind)

2. **Order Executor** (`src/core/execution/order-executor.ts`)
   - Confirms every fill by polling `getOpenOrders()` — fixes the missed-trade bug
   - Auto-cancels resting orders after 6 seconds
   - Returns definitive FILLED/UNFILLED/ERROR — never assumes

3. **Trade Ledger** (`src/core/execution/trade-ledger.ts`)
   - Append-only JSONL log with reconciliation (expected vs actual P&L)
   - Alerts on discrepancies > $0.50

4. **Main Bot** — full pipeline:
   - Monitors candles T-300s to T-35s (builds "never one-sided" history)
   - Evaluates filters at T-30s
   - Executes one trade per candle (best candidate)
   - Confirms fill, waits for resolution, redeems, verifies balance
   - Supports `--filter tight|medium|loose`

### Bug Fix
- Entry window was re-evaluating multiple times per candle (T-32s, T-26s, T-20s, etc.)
- Fixed with `entryEvaluatedForCandle` flag — evaluates once, then waits

## Dry Run Session 1: 0/22 — Finding & Fixing Bot Bugs (Late Evening)

### What happened
- Bot ran 7:30pm-1:30am ET (off-peak hours — the weakest time block)
- **0 wins out of 22 trades** — far worse than any streak in collector data (worst was 5)
- Triggered investigation: is the collector wrong, or is the bot wrong?

### Root cause: Bot was NOT the collector
Cross-referenced collector data for the exact same time window:
- **Collector: 38 trades, 28 wins = 73.7% win rate** during the same period
- **Bot: 22 trades, 0 wins = 0%** during the same period
- Collector is solid. Bot had two bugs.

### Bug 1: Selection algorithm picked the worst trades
- Bot sorted candidates by lowest ask (cheapest underdog) thinking "biggest payout"
- This selected 4-6¢ extreme underdogs that win ~5% of the time
- Collector's wins were at 26-44¢ asks — reasonable underdogs with ~40-50% win rates
- **Fix:** Changed selection to pick candidate closest to 50¢ (best risk/reward)

### Bug 2: No max price guard — bought at 100¢
- Trades 13 & 14 bought "underdog" at $1.00 — actually the favorite at full price
- Caused by stale/empty book confusing underdog detection
- **Fix:** Added hard cap — skip any trade with underdog ask > 50¢

### Bug 3 (from earlier): Entry window re-evaluation
- Bot evaluated the same candle multiple times (T-32s, T-26s, T-20s, etc.)
- **Fix:** `entryEvaluatedForCandle` flag — evaluate once per candle

### Key insight
The 0/22 streak was NOT evidence the strategy is broken. The collector confirmed 74% win rate in the same window. The bot was just systematically picking the worst available trades. This is exactly why you dry run.

## Dry Run Session 2: Selection fix was broken — 0/36 continues

### What happened
- Restarted with "pick closest to 50¢" fix, but went 0/4 more (0/36 cumulative)
- Investigated: trade 36 had SOL UP @19¢ and XRP UP @6¢ — bot picked XRP @6¢
- **The sort was backwards.** `Math.abs(b - 0.50) - Math.abs(a - 0.50)` sorts largest distance first = still picking cheapest
- **Fix:** Flipped to `Math.abs(a - 0.50) - Math.abs(b - 0.50)` = smallest distance first = closest to 50¢

### The real culprit all along
The 0/22 and 0/36 streaks were almost entirely caused by the selection algorithm consistently picking the worst trade from every set of candidates. The 50¢ cap helped block the 100¢ garbage, but the sort order meant it was still choosing 4-6¢ extreme underdogs over 19-33¢ reasonable ones.

## Dry Run Session 3: ETH DOWN @44¢ — UNKNOWN resolution bug

### What happened
- First trade with corrected sort: ETH DOWN @44¢ — correctly picked closest to 50¢
- Resolution came back UNKNOWN — API was too slow (only 2 min retry window)
- Collector confirms it was a LOSS (resolved UP, CL moved +$0.20)
- But the real problem: UNKNOWN was silently counted as a loss in P&L tracking
- **This is the exact same root cause as the previous live max loss bug**

### Bug 4: UNKNOWN resolution handling
- Old behavior: UNKNOWN counted as loss, corrupting P&L tracking
- **Fix:** Extended retries to ~4.5 min total (20s wait + 30 retries, then 30s wait + 20 more retries)
- UNKNOWN trades are now logged but NOT counted in win/loss P&L
- In live mode, falls back to on-chain balance change to infer outcome
- Flagged for manual review

- Bootstrap gives only 3 quick retries per crypto. If API is slow at startup, that crypto never becomes tradeable
