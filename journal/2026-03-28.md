## Session Summary

Revalidated all strategy numbers. Strategy is actually stronger than we thought.

## The Resolution Bug

### Discovery
- Bot showed 16.7% win rate (4/24) while collector showed 53.7% (22/41) in the same window
- Trade-by-trade comparison revealed: 5 out of 9 shared trades had DIFFERENT resolutions
- Same slug, same crypto — bot said DOWN, collector said UP, or vice versa

### Root Cause
- **Gamma API `outcomePrices` disagrees with Chainlink 8% of the time** (729 out of 9,096 candles)
- Collector line 1050: `const resolution = apiResolution !== 'UNKNOWN' ? apiResolution : clResolution` — preferred API over Chainlink
- Bot used `resolveOutcome()` which polls Gamma API exclusively
- These markets resolve based on Chainlink oracles, so **Chainlink is the ground truth**

### Cascading Effect
- Wrong resolution → wrong prev resolution stored → next candle's prev match filter checks against wrong data
- One bad resolution poisons ALL future trades for that crypto
- Bot was taking trades it shouldn't (14 trades collector didn't qualify → 2W/14L = 14%) and missing trades it should (33 trades collector saw → 18W/15L = 55%)

### Fix
- **Collector**: Flipped priority — Chainlink first, API for logging only (line 1050)
- **Bot**: Added `ChainlinkFeed`, records open prices per candle, resolves by comparing CL close vs open
- **Bot**: API only used for bootstrap (startup prev resolution when no CL history)
- **Collector restarted** (PID 94482) with fix
- **Bot restarted** (PID 94169) with fix

## Bug #9: Blocking resolution — bot missed 60% of candles

### Discovery
- After Chainlink fix, bot still only saw 11 of collector's 27 trades
- 10 of 14 missed 5m trades: "No entry window log found — bot in resolution wait"
- Bot was blocking after every trade: wait for candle end (30s), wait for settlement (20s), resolve (5-30s), redeem, balance check
- By the time it finished, 1-2 entire candles had passed

### Fix: Non-blocking resolution queue
- Trades are placed at T-30s then immediately queued as "pending"
- Bot continues to next candle without waiting
- Pending trades are resolved during next candle's monitoring phase (plenty of idle time T-300 to T-35)
- Also fixed Phase 4 `nextBoundary` wait that was sleeping up to 5 minutes after resolution

### Verification
- Watched 2 consecutive cycles: trade at T-33s → queued → resolved 44s later during next monitoring
- No more missed candles — bot evaluates every single 5-minute cycle

## Bug #10: Collector duplicate rows (race condition)
- Every candle was being written TWICE since per-crypto prev restart
- Cause: `Promise.all` batch collection — same slug appeared twice in `activeMarkets`, both passed dedup check simultaneously
- Fix: dedup batch by slug BEFORE `Promise.all`
- Impact: inflated trade counts in analysis (win rates were unaffected)

## Bug #11: CL open timing for 15m candles
- Bot recorded CL open when it first DETECTED a 15m candle (up to 5 min late)
- Collector records CL open at actual candle start
- $0.50+ price shift on ETH caused resolution flips
- Fix: store CL prices at every 5-min boundary in a history buffer. 15m candles look up the correct boundary price from history.

## Bug #12: Gamma API still used for resolution fallback
- `resolveOutcomeAPI` was called as fallback when CL returned UNKNOWN
- Removed all API resolution — Chainlink is the ONLY resolution source
- Gamma API now used ONLY for market discovery (finding active markets + token IDs)
